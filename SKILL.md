# 方案文件修改Skill

## 定位

专为**国内体制内场景**设计的方案文档修改助手，适用于：
- 政府机关公文、项目申报书
- 国有企业项目建议书、可行性研究报告
- 事业单位计划书、预算方案
- 各类需要逻辑严谨、数据一致的正式方案文档

## 适用场景

用户需要对Word文档（doc/docx格式）进行修改，涉及：
- 项目建议书、可行性报告、计划书、预算方案等方案类文档
- 包含数据、逻辑、结论等多维度内容的修改
- 需要保持行文逻辑自洽，避免"贴补丁"式修改

## 工作流程

### 1. 接收任务

**必做**：
- 确认修改范围：用户要求改哪些部分
- 识别是否涉及逻辑联动：改A是否会带动B、C等一起变

**提问示例**：
- "你这次修改涉及几个方面的内容？"
- "除了你提到的这部分，还有没有基于这部分推导出来的其他表述需要一起调？"

### 2. 分析与审查（核心）

**逻辑联动检查**：
- 用户提出修改后，扫描全文找出基于该内容衍生的其他表述
- 包括：基于数据得出的结论、基于前文展开的推论、引用前文的总结等
- 自动同步调整这些关联内容

**数据一致性检查**：
- 数字是否统一（同一个指标在一各处是否一致）
- 口径是否统一（计算逻辑、统计范围是否一致）
- 结论是否与数据匹配

**行文风格与格式统一**：
- 格式统一：修改内容必须遵循原文格式（字体、字号、标点、编号样式等），避免用户返工
- 风格适配：分析原文的行文风格，采用相似的写作方法进行修改
- 禁止浮夸：禁止使用比喻句、夸张修辞、AI式要点罗列（如"首先、其次、最后"堆砌）
- 禁止表情：禁止使用emoji表情、颜文字、符号装饰
- 符合国企行文：言简意赅、严肃专业、逻辑清晰，避免口语化和网络用语

### 3. 高亮标记

**颜色分配规则**：
1. 阅读用户所有修改需求，识别包含几个不同类别
2. 每个类别分配一个颜色（黄色、绿色、蓝色等，优先使用常见高亮色）
3. 同一类别的修改用同一个颜色标记
4. 关联跟随：因逻辑联动而同步调整的内容，跟随其关联的主修改的类别颜色

**示例**：
- 用户修改意见包含3类：A类、B类、C类 → 分别用3个颜色
- A类中某个数据改了，由此影响的A类结论 → 同样用A类的颜色
- 下个人的修改意见包含2类：品牌定位、表达润色 → 2个颜色

**技术实现**：
- 工具兼容：doc/docx格式同时适配WPS和Microsoft Word
- XML级别修改时使用 `<w:shd w:val="clear" w:fill="颜色代码"/>`
- 黄色=FFFF00，绿色=00FF00，蓝色=0000FF，粉色=FF99CC等

### 4. 输出校验

**必做**：
- 核对高亮是否完整：用户要求的修改是否都标记了
- 回溯关键修改的关联影响：主要修改是否带动了其他需要调整的地方

**校验问题示例**：
- "你提到要改'四大问题'为'五大问题'，文中所有引用这个说法的地方都同步了吗？"
- "预算从15万改到60-80万，相关的收益测算和结论也需要调整吧？"

## 输出要求

- 修改完成后，列出所有修改的位置和颜色分类
- 如有需要用户确认的逻辑联动，提前说明
